## 1. 设计理念

### 1.1 "Agent 中的 Agent"：嵌套 Agent 模式

Claude Code 的 Agent 子系统采用了一种**递归嵌套**的架构思想——主 Agent（用户直接交互的那个 Claude Code 实例）可以在执行过程中**启动子 Agent**，而子 Agent 本身也是完整的 Claude Code 实例，拥有独立的 Agentic Loop。

```
┌─────────────────────────────────────────────────────────┐
│  主 Agent (Main Claude Code Instance)                   │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Agentic Loop                                   │    │
│  │  ┌─────────┐  ┌──────────┐  ┌───────────────┐  │    │
│  │  │ 分析任务 │→│ 决定拆分  │→│ 启动子 Agent   │  │    │
│  │  └─────────┘  └──────────┘  └───────┬───────┘  │    │
│  │                                      │          │    │
│  └──────────────────────────────────────┼──────────┘    │
│                                         │               │
│    ┌────────────────────┬───────────────┼────────┐      │
│    │                    │               │        │      │
│    ▼                    ▼               ▼        ▼      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐ ┌─────────┐  │
│  │ 子Agent-1 │  │ 子Agent-2 │  │ 子Agent-3 │ │子Agent-4│  │
│  │ (Explore) │  │ (Plan)   │  │(General) │ │(Custom) │  │
│  │ Haiku模型  │  │ Sonnet   │  │ Sonnet   │ │ Opus    │  │
│  │ 只读工具   │  │ 只读工具  │  │ 完整工具  │ │自定义工具│  │
│  │ worktree-1│  │ 主目录    │  │worktree-2│ │worktree-3│  │
│  └──────────┘  └──────────┘  └──────────┘ └─────────┘  │
└─────────────────────────────────────────────────────────┘
```

这种设计的核心洞察是：**复杂任务天然具有层级结构**。正如人类团队中的技术负责人会把大项目拆分给不同的工程师，主 Agent 也会把复杂任务拆分给专门化的子 Agent。

### 1.2 为什么需要子 Agent

#### (1) 任务分解

大型任务往往无法在单一上下文窗口内完成。通过将任务分解为多个子任务，每个子 Agent 只需关注自己的子问题，避免了上下文膨胀。

```
重构一个大型模块
    ├── 子Agent-1: 分析现有代码结构和依赖关系
    ├── 子Agent-2: 设计新的架构方案
    ├── 子Agent-3: 实现模块 A 的重构
    ├── 子Agent-4: 实现模块 B 的重构
    └── 子Agent-5: 更新测试用例
```

#### (2) 并行执行

多个子 Agent 可以同时运行，大幅提高效率。例如，同时在不同文件中搜索信息、同时修改互不依赖的模块。

#### (3) 资源隔离

每个子 Agent 拥有独立的：
- **上下文窗口**：不会互相污染
- **工具集**：可以限制子 Agent 的能力（如只读）
- **文件系统视图**：通过 git worktree 实现物理隔离
- **模型选择**：简单任务用便宜快速的模型，复杂任务用强大的模型

#### (4) 故障隔离

子 Agent 失败不会直接崩溃主 Agent。主 Agent 可以捕获子 Agent 的错误，决定是否重试或采用替代方案。

### 1.3 类似于操作系统的进程/线程模型

Agent 子系统的设计与操作系统的进程模型高度相似：

| 操作系统概念 | Agent 子系统对应 |
|------------|----------------|
| 主进程 | 主 Agent（用户交互的 Claude Code） |
| 子进程 | 子 Agent（被启动的嵌套实例） |
| fork() | Agent 工具调用，创建新的 Agentic Loop |
| 进程间通信 (IPC) | SendMessage 机制 |
| 进程隔离 | git worktree + 独立上下文 |
| 进程优先级 | 模型选择（Haiku=低优先级快速，Opus=高优先级强力） |
| 后台进程 | `run_in_background` 参数 |
| 进程退出码 | 子 Agent 返回结果/错误状态 |

```
操作系统进程模型                    Agent 子系统模型
─────────────────                  ─────────────────
   init (PID 1)                    主 Agent
      │                               │
   fork()                          Tool: Agent
      │                               │
   ┌──┴──┐                        ┌───┴───┐
   │     │                        │       │
 子进程1  子进程2                  子Agent1  子Agent2
 (独立内存) (独立内存)             (独立上下文) (独立上下文)
 (独立文件描述符)                  (独立工具集)
```

## 2. Agent 类型详解

Claude Code 提供了多种预定义的 Agent 类型，每种类型针对不同的使用场景进行了优化。

### 2.1 类型总览

```
┌─────────────────────────────────────────────────────────────────────┐
│                      Agent 类型体系                                  │
├────────────────┬──────────┬───────────┬──────────┬─────────────────┤
│ general-purpose│ Explore  │   Plan    │cc-guide  │statusline-setup │
│   通用 Agent    │ 快速探索  │ 架构规划   │使用指南   │ 状态栏配置       │
├────────────────┼──────────┼───────────┼──────────┼─────────────────┤
│ 完整工具集      │ 只读工具  │ 只读工具   │ 文档工具  │ 配置工具         │
│ Sonnet 模型    │ Haiku模型 │ Sonnet模型 │ Sonnet   │ Sonnet          │
│ 可读写代码      │ 不修改代码 │ 不修改代码 │ 不修改代码 │ 修改配置文件      │
│ worktree隔离   │ 主目录    │ 主目录     │ 主目录   │ 主目录           │
├────────────────┴──────────┴───────────┴──────────┴─────────────────┤
│                     自定义 Agent (--agents / 配置文件)                │
│              用户可定义任意 Agent 类型，指定工具、模型、提示词           │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.2 general-purpose（通用 Agent）

**用途**：需要完整能力的子任务，如编写代码、修改文件、运行命令等。

**特点**：
- 拥有与主 Agent 几乎相同的**完整工具集**（Bash, Read, Write, Edit, Grep, Glob 等）
- 使用主力模型（默认 Sonnet）
- 通过 **git worktree** 获得独立的代码副本，避免与主 Agent 或其他子 Agent 的文件修改冲突
- 可以在后台运行

**典型场景**：
```
主 Agent: "重构 user 模块并同时更新 payment 模块"
  ├── 子 Agent (general-purpose): 在 worktree-1 中重构 user 模块
  └── 子 Agent (general-purpose): 在 worktree-2 中更新 payment 模块
```

**工具调用示例**（伪代码）：
```javascript
{
  tool: "Agent",
  params: {
    type: "general-purpose",
    prompt: "重构 src/user/ 目录下的所有模块，将 class-based 组件改为 hooks",
    run_in_background: true
  }
}
```

### 2.3 Explore（快速代码探索）

**用途**：快速搜索和理解代码，不做任何修改。

**特点**：
- 使用 **Haiku** 等小模型（速度快、成本低）
- **只读工具集**：仅包含 Read, Grep, Glob, Bash（受限）等
- **不使用 worktree**：因为不修改文件，无需隔离
- 响应速度极快，适合"快速查一下"的场景

**典型场景**：
```
主 Agent: "修复 login 页面的 bug"
  ├── 子 Agent (Explore): "找到所有与 login 相关的文件和函数"  ← 快速，用 Haiku
  ├── 子 Agent (Explore): "查看 auth 模块的 API 接口定义"      ← 快速，用 Haiku
  └── 主 Agent 自己: 基于探索结果，修复 bug                      ← 用 Sonnet
```

**设计哲学**：不是所有任务都需要最强的模型。探索和搜索这类任务，小模型就够了，而且更快更便宜。这是一种**成本感知**的设计。

### 2.4 Plan（架构设计规划）

**用途**：分析代码库结构、设计架构方案、生成实施计划。

**特点**：
- **只读**，绝不修改代码
- 使用 Sonnet 或更强的模型（需要深度理解能力）
- 输出结构化的计划文档
- 可以访问文件系统来理解项目结构

**典型场景**：
```
用户: "帮我规划如何将这个 monolith 拆分成微服务"
  └── 子 Agent (Plan):
      1. 分析现有代码结构
      2. 识别模块边界
      3. 设计服务拆分方案
      4. 输出详细的迁移计划
```

**与 Explore 的区别**：
| 维度 | Explore | Plan |
|------|---------|------|
| 目标 | 找到具体信息 | 产出结构化方案 |
| 模型 | Haiku（小而快） | Sonnet+（强理解） |
| 深度 | 浅层搜索 | 深度分析 |
| 输出 | 事实性回答 | 结构化计划 |

### 2.5 claude-code-guide（Claude Code 使用指南）

**用途**：回答用户关于 Claude Code 本身的使用问题。

**特点**：
- 内置了 Claude Code 的文档和使用知识
- 不修改代码
- 专注于解释功能、用法、配置

### 2.6 statusline-setup（状态栏配置）

**用途**：帮助用户配置终端状态栏（如 tmux, starship 等）以显示 Claude Code 状态信息。

**特点**：
- 专门化的工具集，只操作配置文件
- 了解各种终端和 shell 的配置方式

### 2.7 自定义 Agent

用户可以通过 `--agents` 参数或配置文件定义自己的 Agent 类型，这是 Claude Code 扩展性的重要体现。

**定义方式**：

```jsonc
// .claude/agents.json 或通过 --agents 参数传入
[
  {
    "name": "security-reviewer",
    "description": "专门进行安全审计的 Agent",
    "model": "claude-sonnet-4-20250514",
    "system_prompt": "你是一个安全专家，专注于发现代码中的安全漏洞...",
    "tools": ["Read", "Grep", "Glob", "Bash"],  // 只读工具
    "readonly": true
  },
  {
    "name": "test-writer",
    "description": "专门编写测试用例的 Agent",
    "model": "claude-sonnet-4-20250514",
    "system_prompt": "你是一个测试工程师，专注于编写全面的测试...",
    "tools": ["Read", "Write", "Edit", "Bash", "Grep", "Glob"],
    "use_worktree": true
  }
]
```

**自定义 Agent 的核心字段**：
- `name`：Agent 名称，用于主 Agent 调用
- `description`：描述，帮助主 Agent 决定何时使用此 Agent
- `model`：使用的模型
- `system_prompt`：系统提示词，定义 Agent 的角色和行为
- `tools`：可用的工具列表
- `readonly`：是否只读模式
- `use_worktree`：是否使用 git worktree 隔离

## 3. Agent 执行机制

### 3.1 本质：嵌套的 Claude Code 实例

每个子 Agent 本质上是一个**完整的 Claude Code 运行时实例**。它不是简单的函数调用，而是拥有自己完整 Agentic Loop 的独立实体。

```
┌──────────────────────────────────────────────────────────────┐
│  子 Agent 实例                                                │
│                                                              │
│  ┌──────────────┐                                            │
│  │ System Prompt │  ← 根据 Agent 类型定制                     │
│  └──────┬───────┘                                            │
│         │                                                    │
│  ┌──────▼───────┐    ┌──────────┐    ┌──────────────┐       │
│  │ Agentic Loop │───→│ LLM API  │───→│ Tool 执行引擎 │       │
│  │ (独立循环)    │←───│ (独立模型) │←───│ (独立工具集)  │       │
│  └──────────────┘    └──────────┘    └──────────────┘       │
│         │                                                    │
│  ┌──────▼───────┐                                            │
│  │ 工作目录      │  ← git worktree 或主目录                   │
│  └──────────────┘                                            │
│         │                                                    │
│  ┌──────▼───────┐                                            │
│  │ 执行结果      │  → 返回给主 Agent                          │
│  └──────────────┘                                            │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 独立性四要素

每个 Agent 实例都拥有四个独立维度：

#### (1) 独立的工具集

```javascript
// 伪代码：不同 Agent 类型的工具集配置
const AGENT_TOOLS = {
  "general-purpose": [
    "Bash", "Read", "Write", "Edit", "Grep", "Glob",
    "Agent", "SendMessage"  // 注意：子 Agent 也可以再启动子 Agent！
  ],
  "explore": [
    "Read", "Grep", "Glob", "Bash"  // 只读工具
  ],
  "plan": [
    "Read", "Grep", "Glob", "Bash"  // 只读工具
  ]
};
```

#### (2) 独立的 System Prompt

每种 Agent 类型有专门定制的 system prompt，引导模型扮演不同角色：

```
Explore Agent 的 system prompt 强调：
- "你的任务是快速搜索和理解代码"
- "不要修改任何文件"
- "直接给出答案，不需要冗长的解释"

Plan Agent 的 system prompt 强调：
- "你是一个架构师"
- "分析代码结构，产出结构化方案"
- "不修改代码，只输出计划"
```

#### (3) 独立的模型选择

不同的 Agent 类型使用不同的 LLM 模型（详见第 4 节）。

#### (4) 独立的工作目录

通过 git worktree 机制实现物理文件隔离（详见第 5 节）。

### 3.3 Worktree 隔离

当子 Agent 需要修改文件时（如 general-purpose 类型），Claude Code 会自动创建一个 **git worktree**，为子 Agent 提供独立的代码副本。

启动流程：

```
1. 主 Agent 调用 Agent 工具
2. 系统检查是否需要 worktree（type=general-purpose 且非只读）
3. 如果需要，执行 git worktree add
4. 子 Agent 在 worktree 目录中执行所有操作
5. 子 Agent 完成后，结果返回主 Agent
6. 主 Agent 决定是否合并 worktree 中的更改
7. 清理 worktree
```

### 3.4 后台运行：run_in_background

子 Agent 支持后台运行，主 Agent 不需要等待其完成即可继续执行其他任务。

```
┌────────────────────────────────────────────────────────┐
│  主 Agent 时间线                                        │
│                                                        │
│  ──●──────●──────────────●──────────────●──→           │
│    │      │              │              │              │
│    │      │   启动子Agent-2  继续自己的工作  汇总结果       │
│    │      │   (background)                             │
│    │   启动子Agent-1                                    │
│    │   (background)                                    │
│    │                                                   │
│  分析任务                                               │
│                                                        │
│  子 Agent-1 时间线                                      │
│  ────────●━━━━━━━━━━━━━━━━━━━●────→                    │
│          │  独立执行...       │ 完成                     │
│                                                        │
│  子 Agent-2 时间线                                      │
│  ──────────────●━━━━━━━━━━━━━━━━━━━●────→              │
│                │  独立执行...       │ 完成               │
└────────────────────────────────────────────────────────┘
```

**关键参数**：
```javascript
{
  tool: "Agent",
  params: {
    prompt: "在 worktree 中实现新的 auth 模块",
    run_in_background: true  // 后台运行，不阻塞主 Agent
  }
}
```

### 3.5 Agent 间通信：SendMessage 机制

当子 Agent 在后台运行时，主 Agent 和子 Agent 之间需要一种通信方式。Claude Code 提供了 **SendMessage** 工具实现这一点。

```
┌──────────┐   SendMessage("进度如何？")   ┌──────────┐
│          │ ──────────────────────────→  │          │
│  主 Agent │                              │ 子 Agent  │
│          │ ←──────────────────────────  │          │
└──────────┘   SendMessage("完成80%")     └──────────┘
```

**通信场景**：
- 主 Agent 向子 Agent 发送补充指令
- 子 Agent 向主 Agent 报告进度
- 子 Agent 向主 Agent 请求澄清
- 主 Agent 要求子 Agent 提前终止

### 3.6 完整的生命周期

```
                    Agent 子系统生命周期
                    ═══════════════════

  ┌─────────┐
  │ 创建阶段 │
  └────┬────┘
       │
       ▼
  ┌──────────────────────────────────────┐
  │ 1. 解析 Agent 类型和参数              │
  │ 2. 确定模型、工具集、system prompt    │
  │ 3. 如需要，创建 git worktree          │
  │ 4. 初始化独立的 Agentic Loop          │
  └────────────────┬─────────────────────┘
                   │
                   ▼
  ┌─────────┐
  │ 执行阶段 │
  └────┬────┘
       │
       ▼
  ┌──────────────────────────────────────┐
  │ 5. 子 Agent 开始自己的 Agentic Loop   │
  │    ┌──────────────────────────────┐  │
  │    │ LLM 调用 → 工具执行 → 反馈   │  │
  │    │      ↑                 │     │  │
  │    │      └─────────────────┘     │  │
  │    └──────────────────────────────┘  │
  │ 6. 可通过 SendMessage 与主 Agent 通信 │
  └────────────────┬─────────────────────┘
                   │
                   ▼
  ┌─────────┐
  │ 完成阶段 │
  └────┬────┘
       │
       ▼
  ┌──────────────────────────────────────┐
  │ 7. 子 Agent 产出最终结果              │
  │ 8. 结果返回给主 Agent                 │
  │ 9. 如有 worktree，主 Agent 决定合并   │
  │ 10. 清理 worktree 和临时资源          │
  └──────────────────────────────────────┘
```

## 4. 模型选择策略

### 4.1 核心思想：用合适的模型做合适的事

不是每个任务都需要最强（也最贵最慢）的模型。Claude Code 的模型选择策略本质上是一种**成本-性能-速度的三角权衡**。

```
                    能力强
                     /\
                    /  \
                   /    \
                  / Opus \
                 /────────\
                / Sonnet   \
               /────────────\
              /    Haiku     \
             /________________\
          速度快 ◄──────────► 成本低
```

### 4.2 各模型的定位

| 模型 | 特点 | 适用 Agent 类型 | 成本 | 速度 |
|------|------|----------------|------|------|
| **Haiku** | 小而快，基础理解能力 | Explore | 最低 | 最快 |
| **Sonnet** | 均衡，强编码能力 | General, Plan, 默认 | 中等 | 中等 |
| **Opus** | 最强理解和推理能力 | 复杂自定义任务 | 最高 | 最慢 |

### 4.3 选择策略详解

```
任务到来
   │
   ▼
┌─────────────────┐    是    ┌──────────────┐
│ 只需搜索/查找？  │────────→│ 使用 Haiku    │
└────────┬────────┘         │ (Explore)    │
         │ 否               └──────────────┘
         ▼
┌─────────────────┐    是    ┌──────────────┐
│ 标准编码/分析？  │────────→│ 使用 Sonnet   │
└────────┬────────┘         │ (默认选择)    │
         │ 否               └──────────────┘
         ▼
┌─────────────────┐    是    ┌──────────────┐
│ 复杂推理/架构？  │────────→│ 使用 Opus     │
└────────┬────────┘         │ (最强模型)    │
         │ 否               └──────────────┘
         ▼
┌─────────────────┐
│ 用户自定义模型    │
└─────────────────┘
```

### 4.4 成本-性能权衡实例

假设一个任务："在一个大型项目中找到所有使用了废弃 API 的文件，并逐一更新"

```
步骤 1: 探索（Explore Agent x Haiku）
  - 搜索所有包含废弃 API 的文件
  - 成本：~$0.001（极低）
  - 速度：~2秒
  - 产出：文件列表

步骤 2: 规划（Plan Agent x Sonnet）
  - 分析每个文件的修改策略
  - 成本：~$0.01
  - 速度：~10秒
  - 产出：修改计划

步骤 3: 执行（General Agent x3 并行 x Sonnet）
  - 在各自 worktree 中修改文件
  - 成本：~$0.03
  - 速度：~30秒（并行，否则 ~90秒）
  - 产出：修改后的代码

总计：~$0.041，~42秒
如果全部用 Opus：~$0.20，~120秒
节省：约 5x 成本，约 3x 时间
```

这种**分级模型策略**是 Agent 子系统的重要经济优势。

## 5. Git Worktree 隔离详解

### 5.1 什么是 Git Worktree

Git worktree 是 Git 内置的功能，允许一个仓库同时有**多个工作目录**，每个工作目录可以检出不同的分支。

```bash
# 常规 Git：一个仓库 = 一个工作目录
/my-project/          ← 唯一的工作目录
  ├── .git/           ← Git 数据库
  ├── src/
  └── package.json

# Git Worktree：一个仓库 = 多个工作目录
/my-project/          ← 主工作目录 (main branch)
  ├── .git/           ← Git 数据库（共享）
  ├── src/
  └── package.json

/tmp/worktree-agent-1/ ← worktree 1 (临时分支)
  ├── .git            ← 指向主仓库的链接文件（不是目录）
  ├── src/            ← 独立的文件副本
  └── package.json

/tmp/worktree-agent-2/ ← worktree 2 (另一个临时分支)
  ├── .git            ← 指向主仓库的链接文件
  ├── src/            ← 独立的文件副本
  └── package.json
```

### 5.2 为什么用 Worktree 而不是 Branch

这是一个关键的设计决策。简单切换分支（branch）无法满足并行执行的需求：

| 方案 | 并行修改 | 文件隔离 | 共享 Git 历史 | 创建成本 |
|------|---------|---------|-------------|---------|
| **git branch** | 不可以（同一工作目录） | 不隔离 | 共享 | 几乎为零 |
| **git clone** | 可以 | 完全隔离 | 不共享（独立副本） | 高（复制整个 .git） |
| **git worktree** | 可以 | 文件隔离 | 共享同一个 .git | 低（只复制工作文件） |

```
方案对比图：

git branch（不行）:
  Agent-1 修改 fileA ──┐
                       ├── 冲突！同一个文件系统
  Agent-2 修改 fileA ──┘

git clone（太重）:
  /clone-1/.git  ← 完整复制（可能几个 GB）
  /clone-2/.git  ← 又一份完整复制

git worktree（刚好）:
  /main/.git     ← 唯一的 Git 数据库
  /worktree-1/   ← 轻量链接，只有工作文件
  /worktree-2/   ← 轻量链接，只有工作文件
```

**Worktree 是最佳平衡点**：既有文件隔离以支持并行修改，又共享 Git 历史以保持一致性，且创建成本低。

### 5.3 隔离的好处

#### (1) 互不影响

```
子 Agent-1 (worktree-1):              子 Agent-2 (worktree-2):
  修改 src/auth.js                       修改 src/payment.js
  修改 src/utils.js                      修改 src/utils.js  ← 不冲突！
  运行 npm test                          运行 npm test       ← 独立运行！
```

即使两个 Agent 同时修改了 `utils.js`，因为是在不同的物理目录中，不会直接冲突。冲突解决被推迟到合并阶段，由主 Agent 处理。

#### (2) 可并行修改

多个 Agent 可以同时对代码库进行修改，无需等待锁或排队。

#### (3) 安全回滚

如果某个子 Agent 的修改不理想，直接删除 worktree 即可，不影响主工作目录。

### 5.4 自动清理机制

Claude Code 会在子 Agent 完成（或失败）后自动清理 worktree：

```
子 Agent 完成
     │
     ▼
┌─────────────────────┐    成功    ┌────────────────────┐
│ 检查执行结果         │──────────→│ 提取更改diff       │
└─────────┬───────────┘           │ 应用到主工作目录    │
          │ 失败                  │ 或创建合并提交      │
          ▼                       └────────┬───────────┘
┌─────────────────────┐                    │
│ 记录错误日志         │                    ▼
└─────────┬───────────┘           ┌────────────────────┐
          │                       │ git worktree remove │
          └──────────────────────→│ 清理临时分支        │
                                  │ 释放磁盘空间        │
                                  └────────────────────┘
```

**清理命令**（内部执行）：
```bash
# 移除 worktree
git worktree remove /tmp/cc-worktree-xxxxx --force

# 删除临时分支
git branch -D cc-agent-xxxxx

# 清理可能残留的 worktree 锁
git worktree prune
```

## 6. 并行 Agent 模式

### 6.1 基本模式

多个子 Agent 同时执行不同的子任务，主 Agent 等待全部完成后汇总结果。

```
┌─────────────────────────────────────────────────────────────┐
│                     并行 Agent 执行流程                       │
│                                                             │
│  主 Agent                                                   │
│  ═══════                                                    │
│       │                                                     │
│       │  1. 分析任务，拆分为子任务                              │
│       │                                                     │
│       ├──→ Agent-1 (Explore): 搜索 API 用法     ─────┐      │
│       ├──→ Agent-2 (Explore): 搜索测试文件       ─────┤      │
│       ├──→ Agent-3 (General): 修改模块 A        ─────┤      │
│       └──→ Agent-4 (General): 修改模块 B        ─────┤      │
│                                                      │      │
│       ┌──────────────────────────────────────────────┘      │
│       │  2. 所有子 Agent 完成                                │
│       ▼                                                     │
│       │  3. 汇总结果                                        │
│       │  4. 合并 worktree 更改                               │
│       │  5. 解决可能的冲突                                    │
│       │  6. 输出最终结果                                      │
│       ▼                                                     │
│    完成                                                     │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 类比：MapReduce 模式

并行 Agent 模式与 MapReduce 有着高度的结构相似性：

```
MapReduce                           Agent 并行模式
──────────                          ──────────────

   Input Data                        复杂任务
      │                                 │
   ┌──┴──┐                          ┌───┴───┐
   │Split│                          │ 分解  │
   └──┬──┘                          └───┬───┘
      │                                 │
  ┌───┼───┐                        ┌────┼────┐
  │   │   │                        │    │    │
  ▼   ▼   ▼                        ▼    ▼    ▼
┌──┐┌──┐┌──┐                   ┌────┐┌────┐┌────┐
│M1││M2││M3│  Map              │ A1 ││ A2 ││ A3 │  子 Agent
└┬─┘└┬─┘└┬─┘                   └─┬──┘└─┬──┘└─┬──┘
 │   │   │                       │    │    │
 ▼   ▼   ▼                       ▼    ▼    ▼
┌──┐┌──┐┌──┐                   ┌────┐┌────┐┌────┐
│R1││R2││R3│  结果              │结果1││结果2││结果3│  子任务结果
└┬─┘└┬─┘└┬─┘                   └─┬──┘└─┬──┘└─┬──┘
 │   │   │                       │    │    │
 └───┼───┘                       └────┼────┘
     │                                │
  ┌──┴───┐                       ┌────┴───┐
  │Reduce│                       │  汇总   │
  └──┬───┘                       └────┬───┘
     │                                │
     ▼                                ▼
  最终结果                          最终结果
```

**对应关系**：

| MapReduce | Agent 并行模式 | 说明 |
|-----------|-------------|------|
| Input Split | 任务分解 | 主 Agent 分析任务，拆分子任务 |
| Map Phase | 子 Agent 并行执行 | 每个 Agent 独立处理子任务 |
| Shuffle | 结果收集 | 子 Agent 完成后返回结果 |
| Reduce Phase | 结果汇总 | 主 Agent 合并所有结果 |

## 7. 对比其他多 Agent 框架

### 7.1 框架对比总览

```
┌─────────────┬──────────────┬──────────────┬──────────────┬──────────────┐
│   特性       │ Claude Code  │   CrewAI     │   AutoGen    │  LangGraph   │
│             │ Agent子系统   │              │              │              │
├─────────────┼──────────────┼──────────────┼──────────────┼──────────────┤
│ 架构模型     │ 嵌套递归      │ 角色协作      │ 对话驱动     │ 状态图        │
│ Agent 关系   │ 父子层级      │ 扁平对等      │ 对话群组     │ 节点+边       │
│ 通信方式     │ SendMessage  │ 任务委派      │ 对话消息     │ 状态传递       │
│ 隔离方式     │ git worktree │ 无（共享环境） │ 无           │ 无            │
│ 模型混用     │ 原生支持      │ 支持          │ 支持         │ 支持          │
│ 执行环境     │ 真实文件系统  │ 抽象工具      │ 代码执行器   │ 抽象工具       │
│ 并行执行     │ 原生 worktree │ 需手动编排    │ 有限支持     │ 并行节点       │
│ 适用场景     │ 软件工程任务  │ 通用业务流程  │ 对话式协作   │ 复杂工作流     │
└─────────────┴──────────────┴──────────────┴──────────────┴──────────────┘
```

### 7.2 Claude Code vs CrewAI

**CrewAI** 的核心概念是 "Crew"（团队），多个 Agent 扮演不同角色协作完成任务。

```python
# CrewAI 的方式
crew = Crew(
    agents=[researcher, writer, reviewer],
    tasks=[research_task, writing_task, review_task],
    process=Process.sequential  # 或 Process.hierarchical
)
```

**核心区别**：

| 维度 | Claude Code | CrewAI |
|------|------------|--------|
| Agent 创建 | 运行时动态创建 | 预定义静态配置 |
| 层级关系 | 真正的父子嵌套，子 Agent 可再嵌套 | 扁平或简单的层级 |
| 隔离 | 物理文件系统隔离（worktree） | 逻辑隔离（共享环境） |
| 灵活性 | LLM 自主决定何时启动子 Agent | 预定义工作流 |

**Claude Code 的优势**：灵活性极高，主 Agent 可以根据实际情况动态决定是否需要子 Agent、需要几个、用什么类型。CrewAI 需要预先定义好团队结构。

**CrewAI 的优势**：更适合固定流程的业务场景，工作流清晰可预测。

### 7.3 Claude Code vs AutoGen

**AutoGen**（微软）的核心概念是多 Agent 对话，Agent 之间通过"对话"协作。

```python
# AutoGen 的方式
assistant = AssistantAgent("assistant", llm_config=...)
user_proxy = UserProxyAgent("user_proxy", code_execution_config=...)
user_proxy.initiate_chat(assistant, message="...")
```

**核心区别**：

| 维度 | Claude Code | AutoGen |
|------|------------|---------|
| 协作模式 | 命令式（父指挥子） | 对话式（Agent 间讨论） |
| 通信 | SendMessage（定向） | 群组对话（广播） |
| 适用场景 | 明确的任务执行 | 开放式讨论和探索 |
| 效率 | 高（直接执行） | 可能低（对话开销） |

**Claude Code 的优势**：任务执行效率高，直接分配和执行，不需要多轮对话达成共识。

**AutoGen 的优势**：在需要多视角讨论的场景（如代码审查、方案评估）更自然。

### 7.4 Claude Code vs LangGraph

**LangGraph** 将工作流建模为有向图，Agent 是图中的节点。

```python
# LangGraph 的方式
graph = StateGraph(AgentState)
graph.add_node("researcher", research_agent)
graph.add_node("coder", coding_agent)
graph.add_edge("researcher", "coder")
```

**核心区别**：

| 维度 | Claude Code | LangGraph |
|------|------------|-----------|
| 建模方式 | 隐式（LLM 决策） | 显式（状态图） |
| 流程控制 | 自适应 | 预定义图结构 |
| 可观测性 | 日志 | 图可视化 |
| 复杂流程 | LLM 自行编排 | 条件边、循环等精确控制 |

**Claude Code 的优势**：不需要预定义工作流图，LLM 自主决策更灵活。

**LangGraph 的优势**：工作流完全可控、可观测、可复现，适合需要精确流程控制的生产环境。

### 7.5 Claude Code 的独特之处

Claude Code 的 Agent 子系统有几个独特的设计选择：

```
┌──────────────────────────────────────────────────────────┐
│           Claude Code Agent 子系统的独特设计               │
│                                                          │
│  1. 真正的递归嵌套                                        │
│     └── 子 Agent 可以再创建子 Agent，形成任意深度的树     │
│                                                          │
│  2. 物理文件隔离                                          │
│     └── git worktree 不是模拟隔离，是真正的文件系统隔离    │
│                                                          │
│  3. 成本感知的模型选择                                     │
│     └── 不同任务自动使用不同成本级别的模型                  │
│                                                          │
│  4. LLM 自主编排                                          │
│     └── 不需要预定义工作流，由 LLM 动态决策               │
│                                                          │
│  5. 与真实开发环境深度集成                                 │
│     └── 不是抽象的"工具调用"，是真正的 git/bash/文件操作   │
└──────────────────────────────────────────────────────────┘
```

### 7.6 选型建议

```
需求场景                              推荐框架
────────                              ────────
软件工程任务（读写代码、重构）          → Claude Code Agent 子系统
固定业务流程（审批、数据处理）          → CrewAI / LangGraph
开放式探索和讨论                       → AutoGen
需要精确流程控制的生产环境              → LangGraph
需要快速原型验证                       → CrewAI
需要与真实开发工具链集成               → Claude Code Agent 子系统
```

## 总结

Agent 子系统是 Claude Code 架构中实现**复杂任务自动化**的核心机制。它的设计充分借鉴了操作系统的进程模型，通过嵌套实例化、资源隔离（git worktree）、模型分级和并行执行，实现了一个既灵活又高效的多 Agent 协作系统。

```
Agent 子系统核心价值
════════════════════

  灵活性：LLM 动态决策 → 不需要预定义工作流
     +
  隔离性：git worktree → 安全并行修改
     +
  经济性：模型分级 → 用最少的钱做最多的事
     +
  可扩展性：自定义 Agent → 适应任意场景
     =
  ──────────────────────────────
  强大的软件工程自动化能力
```
