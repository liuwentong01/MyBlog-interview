# Claude Code 架构概览

> 本文是对 Claude Code（Anthropic 官方 CLI Agent）架构的深度学习笔记，旨在从整体视角理解其设计理念、核心架构和运行机制。

---

## 一、Claude Code 是什么

Claude Code 是 Anthropic 官方推出的 **基于 Node.js 的命令行 AI Agent 系统**。它不是一个简单的聊天机器人客户端，而是一个能够自主完成复杂开发任务的智能代理（Agent）。

**核心定位：**

- 它是一个 **CLI 工具**，运行在终端中，深度融入开发者的日常工作流
- 它是一个 **Agent 系统**，能够自主规划、执行多步骤任务，而非简单的一问一答
- 它具备 **本地文件系统操作能力**，可以读写文件、执行命令、搜索代码
- 它通过 **Anthropic Claude API** 获取智能决策能力

**技术栈：**

| 层面 | 技术选型 |
|------|----------|
| 运行时 | Node.js |
| 界面 | Terminal UI（TUI）/ Headless 模式 |
| LLM 通信 | Anthropic Messages API（Streaming） |
| 工具系统 | 结构化 Tool Calling |
| 打包 | 单文件 Bundle（esbuild） |

简单来说，Claude Code = **CLI 外壳 + Agent 内核 + Tool 生态 + 权限安全模型**。

## 三、四大设计理念详解

### 3.1 Agent Loop 模式

**传统聊天模式：**

```
用户提问 → LLM 回答 → 结束
```

**Agent Loop 模式：**

```
用户提问 → LLM 思考 → 调用工具A → 获取结果 → LLM 再思考 → 调用工具B
→ 获取结果 → LLM 再思考 → 调用工具C → ... → LLM 认为任务完成 → 返回文本
```

这是 Claude Code 最核心的设计模式。Agent Loop 的本质是：**LLM 不仅是回答者，更是决策者。它决定下一步做什么、调用什么工具、何时结束。**

**Agent Loop 伪代码：**

```javascript
async function agenticLoop(userMessage) {
  // 初始消息列表
  const messages = [...history, { role: 'user', content: userMessage }];

  while (true) {
    // 1. 调用 LLM API
    const response = await callLLM(messages);

    // 2. 收集响应中的所有内容块
    for (const block of response.content) {
      if (block.type === 'text') {
        // 文本块 → 流式输出给用户
        displayToUser(block.text);
      }

      if (block.type === 'tool_use') {
        // 3. 工具调用块 → 权限检查 → 执行
        const permitted = await checkPermission(block.name, block.input);
        if (!permitted) {
          // 用户拒绝 → 记录拒绝结果
          messages.push({
            role: 'tool',
            content: 'Permission denied by user',
            tool_use_id: block.id
          });
          continue;
        }

        const result = await executeTool(block.name, block.input);

        // 4. 将工具结果加入消息历史
        messages.push({
          role: 'assistant',
          content: [block]  // tool_use 块
        });
        messages.push({
          role: 'user',
          content: [{
            type: 'tool_result',
            tool_use_id: block.id,
            content: result
          }]
        });
      }
    }

    // 5. 如果没有工具调用，说明 LLM 认为任务完成
    const hasToolUse = response.content.some(b => b.type === 'tool_use');
    if (!hasToolUse) {
      break;  // 退出循环
    }

    // 否则继续循环，让 LLM 根据工具结果继续推理
  }
}
```

**为什么这个模式如此强大？**

- **多步推理**：复杂任务（如重构代码）需要：先搜索相关文件 → 读取文件内容 → 理解代码 → 编辑文件 → 验证修改。单次问答无法完成。
- **自主纠错**：如果某步操作失败（如 bash 命令报错），LLM 会看到错误信息，自动调整策略。
- **动态规划**：LLM 不需要预先知道所有步骤，它根据每一步的结果动态决定下一步。

**一个真实执行链的例子：**

用户说："修复这个 TypeScript 编译错误"

```
[Loop 1] LLM → 调用 Bash: "npx tsc --noEmit 2>&1"
                → 获取编译错误信息

[Loop 2] LLM → 调用 Read: 读取出错的文件
                → 获取文件内容

[Loop 3] LLM → 调用 Grep: 搜索相关类型定义
                → 找到类型定义位置

[Loop 4] LLM → 调用 Edit: 修改文件修复类型错误
                → 文件已修改

[Loop 5] LLM → 调用 Bash: "npx tsc --noEmit 2>&1"
                → 确认编译通过

[Loop 6] LLM → 输出文本: "已修复 TypeScript 编译错误，问题是..."
                → 循环结束
```

### 3.2 工具驱动（Tool-Driven Architecture）

Claude Code 的所有外部操作都通过 **结构化的 Tool 系统** 完成。LLM 本身不能直接操作文件系统或执行命令——它只能通过请求调用工具来间接操作。

**核心工具集：**

| 工具名 | 功能 | 典型用途 |
|--------|------|----------|
| `Bash` | 执行 shell 命令 | 运行测试、安装依赖、git 操作 |
| `Read` | 读取文件内容 | 查看源代码、配置文件 |
| `Write` | 写入/创建文件 | 创建新文件、完全重写文件 |
| `Edit` | 编辑文件（diff） | 修改现有文件的部分内容 |
| `Grep` | 正则搜索文件内容 | 在代码库中查找模式 |
| `Glob` | 文件名模式匹配 | 查找特定类型的文件 |
| `WebFetch` | 获取网页内容 | 查阅文档、下载资源 |

**工具调用的协议格式：**

LLM 和系统之间通过 Anthropic Messages API 的 Tool Use 协议通信：

```json
// LLM 返回的工具调用请求
{
  "type": "tool_use",
  "id": "toolu_01abc123",
  "name": "Bash",
  "input": {
    "command": "find /project -name '*.ts' -type f | head -20"
  }
}

// 系统返回的工具执行结果
{
  "type": "tool_result",
  "tool_use_id": "toolu_01abc123",
  "content": "src/index.ts\nsrc/utils.ts\nsrc/types.ts\n..."
}
```

**为什么选择工具驱动而不是直接执行？**

1. **可审计性**：每次操作都有明确的输入和输出记录
2. **可控性**：系统可以在执行前拦截、审查、拒绝任何操作
3. **可扩展性**：添加新能力只需注册新工具，不改变核心循环
4. **标准化**：工具定义遵循统一的 JSON Schema，LLM 天然理解

**工具注册的概念模型：**

```javascript
// 每个工具都是一个结构化定义
const BashTool = {
  name: 'Bash',
  description: '执行 shell 命令并返回输出',
  parameters: {
    type: 'object',
    properties: {
      command: {
        type: 'string',
        description: '要执行的 shell 命令'
      }
    },
    required: ['command']
  },
  // 实际执行函数
  async execute({ command }) {
    const result = await exec(command);
    return { stdout: result.stdout, stderr: result.stderr };
  }
};
```

### 3.3 权限控制（Permission Model）

Claude Code 的权限系统是一个重要的安全特性。**并非所有工具调用都被自动允许**——破坏性操作需要用户明确授权。

**权限分类：**

```
┌─────────────────────────────────────────────────┐
│              权限分类模型                         │
│                                                  │
│  ┌──────────────────┐                           │
│  │ 只读操作 (低风险) │  ← 通常自动允许            │
│  │ Read, Grep, Glob │                           │
│  └──────────────────┘                           │
│                                                  │
│  ┌──────────────────┐                           │
│  │ 写入操作 (中风险) │  ← 需要确认或配置允许       │
│  │ Write, Edit      │                           │
│  └──────────────────┘                           │
│                                                  │
│  ┌──────────────────┐                           │
│  │ 执行操作 (高风险) │  ← 需要用户明确确认         │
│  │ Bash             │                           │
│  └──────────────────┘                           │
└─────────────────────────────────────────────────┘
```

**权限检查流程：**

```javascript
async function checkPermission(toolName, toolInput) {
  // 1. 检查是否在白名单中（用户预先允许的操作）
  if (isInAllowList(toolName, toolInput)) {
    return true;  // 自动允许
  }

  // 2. 检查是否是只读操作
  if (isReadOnlyTool(toolName)) {
    return true;  // 自动允许
  }

  // 3. 需要用户确认
  const userDecision = await promptUser(
    `是否允许执行 ${toolName}？\n输入: ${JSON.stringify(toolInput)}`
  );

  // 4. 用户可以选择：允许一次 / 始终允许 / 拒绝
  return userDecision.allowed;
}
```

**权限配置的层级：**

```
全局配置 (~/.claude/settings.json)
    └── 项目配置 (.claude/settings.json)
        └── CLAUDE.md 中的指令
            └── 运行时用户确认
```

这种多层级的权限模型确保了：
- **安全性**：不会在用户不知情的情况下执行危险命令（如 `rm -rf`）
- **效率**：常用的安全操作可以配置为自动允许，避免频繁确认
- **灵活性**：不同项目可以有不同的权限策略

### 3.4 双模式运行

Claude Code 支持两种运行模式，适应不同使用场景：

#### 交互式 TUI 模式（Terminal User Interface）

```bash
# 直接启动，进入交互式界面
claude
```

- 用户在终端中与 Claude 进行多轮对话
- 实时显示 Agent 的思考过程和工具调用
- 工具调用时弹出权限确认提示
- 支持 `/` 开头的斜杠命令（如 `/help`, `/clear`）
- 适合日常开发中的探索性任务

#### Headless 模式（非交互/编程式调用）

```bash
# 通过 -p 参数传入提示，非交互执行
claude -p "查找所有未使用的依赖并列出"

# 输出 JSON 格式结果
claude -p "分析代码结构" --output-format json
```

- 没有交互式 UI，从命令行参数或 stdin 读取输入
- 输出结构化结果（文本或 JSON）
- 适合 CI/CD 流水线、脚本自动化、与其他工具集成
- 权限策略通常预先配置为自动允许

**两种模式的对比：**

| 特性 | TUI 模式 | Headless 模式 |
|------|----------|---------------|
| 用户交互 | 多轮对话 | 单次输入/输出 |
| 权限确认 | 实时弹窗 | 预配置策略 |
| 输出格式 | 富文本终端渲染 | 纯文本 / JSON |
| 使用场景 | 开发调试 | CI/CD 自动化 |
| 启动方式 | `claude` | `claude -p "..."` |

## 四、与传统 CLI 工具的区别

这是理解 Claude Code 的关键：**它和你用过的任何传统 CLI 工具都不一样。**

### 传统 CLI 工具：确定性管道

```bash
# 传统 CLI：每个命令都是确定性的
grep -r "TODO" src/          # 输入确定 → 输出确定
sed -i 's/foo/bar/g' file.js # 输入确定 → 操作确定
npm test                     # 执行确定的测试套件
```

传统 CLI 的特点：
- **确定性**：相同输入总是产生相同输出
- **单步执行**：一个命令做一件事
- **用户驱动**：每一步都由用户决定
- **管道组合**：通过 `|` 组合多个命令

### Agent CLI：非确定性循环

```bash
# Agent CLI：用自然语言描述意图，Agent 自主执行
claude -p "重构 UserService，将数据库访问抽出为 Repository 模式"
```

Agent CLI 的特点：
- **非确定性**：相同输入可能产生不同的执行路径（取决于 LLM 的推理）
- **多步自主执行**：Agent 自己决定需要几步、每步做什么
- **Agent 驱动**：用户提出目标，Agent 自主规划和执行
- **循环而非管道**：执行模式是循环，不是线性管道

**对比图：**

```
传统 CLI:
  命令1 → 输出1 → 命令2 → 输出2 → 命令3 → 输出3
  (每步由用户手动驱动)

Agent CLI:
  用户意图 → [Agent Loop: 分析 → 工具1 → 分析 → 工具2 → ... → 完成]
  (由 LLM 自主驱动)
```

**这意味着什么？**

1. **不可预测的执行时间**：Agent 可能只需要一步，也可能需要二十步
2. **不可预测的资源消耗**：每轮循环都消耗 API tokens
3. **需要安全保障**：因为行为不可完全预测，所以权限系统至关重要
4. **强大的问题解决能力**：可以处理模糊的、需要探索的任务

## 五、代码规模分析

Claude Code 的发布形态是一个 **单文件 JavaScript Bundle**，通过 esbuild 将所有源代码打包为一个文件。

**规模数据：**

```
单文件 Bundle 规模
├── 代码行数:    ~15,600 行
├── Token 数:    ~197,000 tokens (约 197K)
├── 打包工具:    esbuild
└── 运行时:      Node.js
```

**这个规模意味着什么？**

- 约 197K tokens 的代码量，表明这是一个 **中大规模** 的应用
- 作为对比，一个典型的中型 Web 应用前端打包后约 50K-200K 行
- 单文件 Bundle 的设计使得部署极其简单：`npm install -g @anthropic-ai/claude-code` 即可

**核心模块的估算分布：**

```
┌─────────────────────────────────────┐
│         代码规模分布（估算）          │
│                                     │
│  Agentic Loop 核心    ████░░  ~15%  │
│  工具定义与执行       ██████░  ~25%  │
│  TUI 渲染与交互       █████░░  ~20%  │
│  权限与安全系统       ███░░░░  ~10%  │
│  会话与消息管理       ███░░░░  ~10%  │
│  配置与 CLI 解析      ██░░░░░  ~8%   │
│  LLM API 通信        ██░░░░░  ~7%   │
│  其他（工具函数等）   █░░░░░░  ~5%   │
└─────────────────────────────────────┘
```

值得注意的是，工具相关的代码占比最大（约 25%），这印证了 Claude Code **工具驱动** 的设计理念——大量代码用于定义工具的行为、参数校验、结果处理和错误恢复。

## 六、与其他 Agent 框架的对比

市面上有多种 Agent 框架，理解 Claude Code 的差异化定位有助于更深入地理解其设计选择。

### 对比总览

| 维度 | Claude Code | LangChain | AutoGPT | Open Interpreter |
|------|------------|-----------|---------|-----------------|
| **定位** | 开发工具 | 通用框架 | 自主 Agent | 代码解释器 |
| **运行环境** | 本地终端 | 应用服务端 | 独立运行 | 本地终端 |
| **LLM 绑定** | Claude Only | 多 LLM | 多 LLM | 多 LLM |
| **工具系统** | 内置开发工具 | Chain/Tool 抽象 | Plugin 系统 | 代码执行 |
| **权限模型** | 多层级安全 | 无内置 | 基础限制 | 用户确认 |
| **目标用户** | 开发者 | 框架开发者 | 技术爱好者 | 开发者 |

### Claude Code 的三大差异化特点

#### 1. 深度集成到开发工作流

```
LangChain / AutoGPT:
  通用 Agent → 可以做很多事 → 但什么都不精

Claude Code:
  专注开发 → 理解代码库结构 → 理解 git 工作流 → 理解项目配置
```

Claude Code 不追求"万能"，它专注于软件开发这一个垂直领域，并做到极致：

- **内置 CLAUDE.md 机制**：每个项目可以定义自己的指令，让 Agent 理解项目约定
- **Git 感知**：自动读取 git 状态、分支信息，理解代码的变更上下文
- **代码库全局搜索**：通过 Grep/Glob 工具快速导航大型代码库
- **增量编辑而非全量重写**：Edit 工具只发送 diff，尊重开发者对代码的所有权

#### 2. 本地文件系统操作

```
多数 Agent 框架:
  运行在服务端/云端 → 通过 API 间接操作 → 延迟高，受限多

Claude Code:
  运行在本地终端 → 直接操作本地文件系统 → 零延迟，全权限
```

这是一个看似简单但影响深远的设计选择：

- **直接读写本地文件**：不需要上传下载，不受文件大小限制
- **直接执行 Shell 命令**：可以运行项目的构建、测试、部署脚本
- **实时感知文件变化**：通过 Glob/Grep 实时搜索最新的代码状态
- **本地 git 操作**：直接执行 git 命令，创建分支、提交、推送

#### 3. 权限安全模型

```
AutoGPT:
  "让 AI 自主运行" → 缺乏安全边界 → 可能造成意外损害

LangChain:
  框架层面不关心安全 → 安全由开发者自行实现

Claude Code:
  内置多层权限模型 → 每个操作可审查 → 用户始终保持控制权
```

这可能是 Claude Code 最独特的设计理念。它试图在两个极端之间找到平衡：

```
完全手动 ◄────────────── Claude Code ──────────────► 完全自主
(效率低)               (有监督的自主)                  (风险高)
```

- **不是无监督的自主 Agent**：每个破坏性操作都经过权限检查
- **也不是被动的问答工具**：它能自主规划和执行多步骤任务
- **"有监督的自主"模式**：在安全边界内，给予 Agent 最大的自由度

---

## 总结

Claude Code 的架构设计可以用一句话概括：

> **一个运行在本地终端的、以工具调用驱动的、有权限安全保障的 Agentic Loop 系统，专为软件开发工作流设计。**

```
核心公式：

Claude Code = CLI 外壳
            + Agentic Loop 内核
            + 结构化 Tool 系统
            + 多层权限模型
            + 开发工作流集成
```

理解了这五个组成部分，就理解了 Claude Code 的全貌。后续的学习笔记将深入每个模块的具体实现。
