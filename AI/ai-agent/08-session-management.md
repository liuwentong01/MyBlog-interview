# 08 - 会话管理与上下文压缩

## 概述

会话管理是 Agent 系统中的基础设施层。与传统 Web 应用中的 Session 管理不同，Agent 的会话管理不仅要维护用户身份和状态，更核心的挑战在于 **如何在有限的 context window 中，跨多轮对话保持足够的上下文信息**，使 Agent 能够持续理解任务并做出正确决策。

---

## 1. 为什么需要会话管理

### 1.1 Agent 任务天然跨越多轮对话

Agent 不是简单的问答系统。一个典型的任务可能需要：

```
用户: "帮我重构这个模块的错误处理"
  → Agent 读取文件，理解现有结构           (第 1 轮)
  → Agent 提出重构方案                    (第 2 轮)
  → 用户反馈需要调整                      (第 3 轮)
  → Agent 修改代码并执行测试               (第 4 轮)
  → 测试失败，Agent 分析错误并修复         (第 5 轮)
  → 全部测试通过，任务完成                 (第 6 轮)
```

在这个过程中，每一轮的决策都依赖于之前所有轮次积累的上下文。如果没有会话管理，Agent 在第 5 轮时将完全不知道自己在做什么。

### 1.2 上下文保持的重要性

上下文不仅仅是"聊天记录"，它包含了：

- **任务目标**：用户最初提出的需求
- **中间状态**：Agent 已经完成了哪些步骤，还剩什么
- **工具执行结果**：读取的文件内容、搜索结果、命令输出
- **决策历史**：为什么选择了方案 A 而不是方案 B
- **环境信息**：当前工作目录、文件结构、项目配置

丢失任何一部分，都可能导致 Agent 重复操作、做出矛盾决策，甚至无法完成任务。

### 1.3 类比：浏览器的 Session vs Agent 的 Session

| 维度 | 浏览器 Session | Agent Session |
|------|---------------|---------------|
| **存储内容** | 用户登录态、偏好设置、购物车 | 完整的消息历史、工具调用结果、任务状态 |
| **数据量** | 通常几 KB | 可能达到数百 KB 甚至 MB（对应数万 tokens） |
| **生命周期** | 浏览器关闭或超时失效 | 任务完成或用户显式结束 |
| **核心挑战** | 安全性（防劫持、防伪造） | 容量限制（context window）和信息保真度 |
| **恢复机制** | Cookie / Session ID | Session ID + 消息历史反序列化 |

浏览器 Session 的难题是安全性，Agent Session 的难题是 **如何在有限的窗口中塞进最重要的信息**。

## 2. 会话持久化机制

### 2.1 整体架构

```
┌─────────────────────────────────────────────────┐
│                  SessionManager                  │
├─────────────────────────────────────────────────┤
│  createSession()    → 创建新会话，生成唯一 ID    │
│  saveSession()      → 序列化并写入存储           │
│  loadSession(id)    → 从存储加载并反序列化        │
│  listSessions()     → 列出所有可用会话           │
│  getLatestSession() → 获取最近活跃的会话          │
│  forkSession(id)    → 从指定会话创建分支          │
├─────────────────────────────────────────────────┤
│               Storage Backend                    │
│         (本地文件系统 / SQLite / etc.)            │
└─────────────────────────────────────────────────┘
```

### 2.2 本地文件系统存储

Claude Code 将会话数据持久化到本地文件系统，典型的存储路径结构：

```
~/.claude/
├── sessions/
│   ├── abc123-def456-789/
│   │   ├── session.json          # 会话元数据
│   │   └── messages.json         # 完整消息历史
│   ├── xyz789-abc123-456/
│   │   ├── session.json
│   │   └── messages.json
│   └── ...
├── session-latest.json           # 指向最近会话的指针
└── config.json                   # 全局配置
```

这种设计的优点：
- **无需数据库依赖**：CLI 工具保持轻量
- **易于调试**：直接查看 JSON 文件即可了解会话状态
- **天然隔离**：每个会话独立目录，不会互相干扰
- **易于清理**：删除目录即删除会话

### 2.3 Session ID 标识

每个会话通过唯一的 Session ID 标识，通常使用 UUID v4 生成：

```typescript
// 伪代码示意
interface Session {
  id: string;                    // UUID v4，如 "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
  createdAt: number;             // 创建时间戳
  updatedAt: number;             // 最后更新时间戳
  cwd: string;                   // 会话关联的工作目录
  model: string;                 // 使用的模型
  parentSessionId?: string;      // fork 来源（如果是分支会话）
  metadata: {
    totalTokens: number;         // 累计消耗的 tokens
    messageCount: number;        // 消息数量
    taskSummary?: string;        // 任务摘要
  };
}
```

Session ID 的用途：
- 唯一标识一次对话
- 作为文件系统中的目录名
- 供 `--resume` 参数引用
- 建立 fork 关系（父子会话）

### 2.4 消息历史序列化

消息历史是会话的核心数据，包含完整的对话流和工具调用记录：

```typescript
// 消息历史的序列化结构
interface SerializedMessage {
  role: "user" | "assistant" | "system";
  content: string | ContentBlock[];   // 文本或结构化内容块
  timestamp: number;
  // 工具调用相关
  tool_calls?: ToolCall[];            // Assistant 发起的工具调用
  tool_call_id?: string;             // tool_result 消息关联的调用 ID
}

interface ToolCall {
  id: string;
  type: "function";
  function: {
    name: string;                     // 如 "Read", "Bash", "Edit"
    arguments: string;                // JSON 字符串化的参数
  };
}
```

序列化时需要处理的特殊情况：
- **大型工具输出**：文件读取结果可能很大，需要考虑是否裁剪
- **二进制内容**：图片等需要 Base64 编码或只保留引用路径
- **敏感信息**：API Key、密码等需要脱敏处理

```
序列化流程:
messages (内存) → JSON.stringify → 压缩(可选) → 写入文件

反序列化流程:
读取文件 → 解压(可选) → JSON.parse → messages (内存)
```

## 3. 会话恢复方式

Claude Code 提供了多种灵活的会话恢复机制，适用于不同的使用场景。

### 3.1 `--resume <session_id>`: 按 ID 恢复指定会话

```bash
claude --resume a1b2c3d4-e5f6-7890-abcd-ef1234567890
```

**工作流程：**

```
1. 解析命令行参数，提取 session_id
2. 从存储中查找对应的会话数据
3. 反序列化消息历史
4. 将历史消息加载到 Agentic Loop 的消息列表中
5. 恢复会话的工作目录（cwd）
6. 等待用户输入新消息，继续对话
```

**适用场景：**
- 精确恢复某个特定的历史会话
- 在不同终端窗口间切换同一任务
- 回到一个之前中断的复杂任务

**注意事项：**
- 加载的是完整的历史消息，会立即占用 context window
- 如果历史消息过多，可能需要在恢复后立即触发压缩

### 3.2 `--continue`: 继续最近的会话

```bash
claude --continue
```

**工作流程：**

```
1. 读取 session-latest 指针（或按 updatedAt 排序查找）
2. 获取最近活跃的 session_id
3. 等价于执行 --resume <latest_session_id>
```

**适用场景：**
- 最常见的恢复方式 —— "我刚才的对话被中断了，继续"
- 终端意外关闭后快速恢复
- 短暂离开后回来继续工作

这是最符合用户直觉的恢复方式，无需记住或查找 Session ID。

### 3.3 `--fork-session`: Fork 一个会话分支

```bash
# fork 当前最近的会话
claude --fork-session

# fork 指定会话
claude --fork-session --resume a1b2c3d4
```

**工作流程：**

```
1. 加载源会话的完整消息历史
2. 创建一个新的 Session ID
3. 深拷贝消息历史到新会话
4. 记录 parentSessionId 指向源会话
5. 源会话保持不变，新会话独立演进
```

**类比 Git 分支：**

```
原始会话:  M1 → M2 → M3 → M4 → M5
                          ↓
Fork 会话:  M1 → M2 → M3 → M6 → M7 (不同的方向)
```

**适用场景：**
- 想尝试不同的解决方案，但不想破坏当前进度
- 对话到了一个关键决策点，想分别探索两个方向
- 需要保留一个"检查点"，可以随时回退
- 类似 Git 的分支开发模式

### 3.4 `--teleport`: 远程会话传输

```bash
# 在机器 A 上导出会话
claude --teleport export > session-dump.json

# 在机器 B 上导入会话
claude --teleport import < session-dump.json
```

**工作流程：**

```
导出端:
  1. 序列化完整会话状态（消息历史 + 元数据 + 环境信息）
  2. 打包为可传输格式
  3. 可选加密

导入端:
  1. 解析传输数据
  2. 创建新的本地会话
  3. 适配本地环境（如路径映射）
  4. 恢复对话上下文
```

**适用场景：**
- 在不同机器间迁移工作上下文
- 团队成员间共享对话进度（"我调试到这一步了，你接着看"）
- 从远程开发机同步会话到本地
- 备份重要的对话历史

**技术挑战：**
- 路径映射：源机器的 `/home/alice/project` 在目标机器上可能是 `/Users/bob/project`
- 环境差异：不同机器的工具集、权限配置可能不同
- 数据安全：传输过程中的加密和验证

### 3.5 恢复方式对比总结

| 方式 | 命令 | 创建新会话 | 适用场景 |
|------|------|-----------|---------|
| Resume | `--resume <id>` | 否 | 精确恢复特定会话 |
| Continue | `--continue` | 否 | 快速恢复最近会话 |
| Fork | `--fork-session` | 是 | 在会话基础上分支探索 |
| Teleport | `--teleport` | 是 | 跨机器迁移会话 |

## 4. 上下文压缩（Context Compression）—— 核心难题

### 4.1 为什么需要压缩

LLM 的 context window 是有限资源。以 Claude 为例：

```
Context Window 容量:  ~200K tokens

一次典型的 Agent 会话消耗:
  System Prompt:           ~3K tokens
  工具定义:                ~5K tokens
  用户消息 (10 轮):        ~2K tokens
  Agent 回复 (10 轮):      ~5K tokens
  工具调用结果 (读文件等):  ~50-150K tokens  ← 大头在这里!
  ─────────────────────────────────
  总计:                    65K - 165K tokens
```

可以看到，**工具调用结果（尤其是文件读取）是 token 消耗的主要来源**。一个中等复杂度的任务，在 10-20 轮对话后就可能逼近 context window 的上限。

如果不做任何处理，超出 context window 后：
- API 调用会直接报错
- 即使不报错，末尾信息的注意力也会衰减
- token 成本线性增长，费用飙升

### 4.2 何时触发压缩

压缩不是每轮都触发的，而是在特定条件下自动启动：

```typescript
// 伪代码：压缩触发逻辑
function shouldCompress(messages: Message[], config: Config): boolean {
  const currentTokens = countTokens(messages);
  const threshold = config.contextWindow * config.compressionThreshold;
  // 例如: 200K * 0.8 = 160K

  return currentTokens >= threshold;
}
```

**触发策略：**

```
Token 使用量
│
│  ████████████████████████████████████████  200K (Context Window 上限)
│  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓      160K (压缩触发阈值 ~80%)
│  ░░░░░░░░░░░░░░░░░░░░░░░░░░                安全区域
│  ░░░░░░░░░░░░░░░░░░░░
│  ░░░░░░░░░░░░░░
│  ░░░░░░░░░░
│  ░░░░░░
│  ░░░
└──────────────────────────── 对话轮次 →
   1    5    10   15   20
```

当 token 使用量达到阈值时，系统自动触发压缩，将 token 使用量降回安全区域。

### 4.3 compact_boundary 系统消息

Claude Code 使用一个特殊的系统消息 `compact_boundary` 来标记压缩点：

```typescript
// 压缩时插入的边界标记
const compactBoundary: Message = {
  role: "system",
  content: "[compact_boundary] 以下是对之前对话的压缩摘要...",
  metadata: {
    type: "compact_boundary",
    compressedAt: Date.now(),
    originalMessageCount: 45,     // 被压缩的原始消息数量
    originalTokenCount: 120000,   // 被压缩前的 token 数量
    compressedTokenCount: 8000,   // 压缩后的 token 数量
  }
};
```

**compact_boundary 的作用：**

1. **分隔标记**：清晰地将"压缩摘要"和"后续原始对话"分开
2. **元数据记录**：记录压缩信息，方便调试和审计
3. **多次压缩支持**：长对话可能触发多次压缩，每次都有对应的 boundary

```
压缩前的消息流:
  [System Prompt]
  [User: M1] [Assistant: R1] [Tool: T1]
  [User: M2] [Assistant: R2] [Tool: T2]
  ... (大量历史消息) ...
  [User: M20] [Assistant: R20]

压缩后的消息流:
  [System Prompt]
  [compact_boundary: 摘要 M1-M15 的关键信息]
  [User: M16] [Assistant: R16] [Tool: T16]
  ... (保留最近的原始消息) ...
  [User: M20] [Assistant: R20]
```

### 4.4 压缩策略

Claude Code 使用 **摘要压缩（Summary Compression）** 策略，核心思路是让 LLM 自身来总结之前的对话：

```
压缩流程:

1. 选择压缩范围
   ├── 保留: System Prompt（永远不压缩）
   ├── 压缩: 早期的对话消息（M1 ~ Mn）
   └── 保留: 最近的若干轮对话（Mn+1 ~ Mlatest）

2. 构造压缩 Prompt
   "请总结以下对话的关键信息，保留:
    - 用户的原始目标和需求
    - 已完成的关键操作和结果
    - 当前任务的进展状态
    - 重要的文件路径和代码变更
    - 尚未解决的问题"

3. 调用 LLM 生成摘要
   输入: 需要压缩的历史消息
   输出: 结构化的摘要文本

4. 替换历史消息
   [被压缩的消息] → [compact_boundary + 摘要]
```

**压缩 Prompt 设计的关键点：**

```typescript
const compressionPrompt = `
请将以下对话历史压缩为简洁的摘要。必须保留：
1. 用户的根本目标
2. 已执行的文件操作（创建、修改、删除了哪些文件）
3. 关键的代码变更内容
4. 遇到的错误和解决方式
5. 当前任务的状态（完成了什么，还差什么）
6. 用户明确表达的偏好和约束

可以省略：
- 文件的完整内容（只保留关键片段）
- 中间探索过程中被放弃的方案
- 重复的工具调用结果
- 格式化和排版细节
`;
```

### 4.5 信息损失的权衡

压缩必然伴随信息损失。关键在于**有策略地选择保留什么、丢弃什么**：

**必须保留的信息（高价值）：**

| 信息类型 | 原因 |
|---------|------|
| 用户的核心需求 | 这是整个任务的锚点，丢失意味着方向迷失 |
| 已执行的变更 | 避免重复操作或产生矛盾的修改 |
| 关键错误和解决方案 | 防止重蹈覆辙 |
| 当前进展状态 | Agent 需要知道"做到哪了" |
| 文件路径和项目结构 | Agent 的"空间记忆" |

**可以丢弃的信息（低价值）：**

| 信息类型 | 原因 |
|---------|------|
| 文件完整内容 | 可以重新读取 |
| 中间搜索过程 | 已经找到目标后，搜索路径不重要 |
| 被否定的方案细节 | 只需记住"方案 B 因为 X 原因被否决" |
| 命令的完整输出 | 保留关键结论即可 |
| 重复确认的对话 | "好的，我来执行" 之类的无信息量内容 |

**信息损失矩阵：**

```
                 高频引用        低频引用
              ┌──────────────┬──────────────┐
  高信息密度  │  必须保留     │  尽量保留     │
              │  (用户目标)   │  (错误细节)   │
              ├──────────────┼──────────────┤
  低信息密度  │  可以摘要     │  可以丢弃     │
              │  (文件路径)   │  (中间输出)   │
              └──────────────┴──────────────┘
```

### 4.6 对 Agent 行为的影响

压缩后，Agent 的行为可能出现以下变化：

**1. 重复操作**
```
压缩前: Agent 记得已经读过 fileA.ts
压缩后: 摘要中可能只有 "已分析相关文件"
结果:   Agent 可能再次读取 fileA.ts
```

**2. 决策偏移**
```
压缩前: Agent 记得用户说 "不要用 class 语法"
压缩后: 这个偏好可能在摘要中被遗漏
结果:   Agent 可能生成使用 class 的代码
```

**3. 上下文断裂感**
```
压缩前: 连续的对话流，逻辑清晰
压缩后: "之前讨论了 X"（摘要）→ 新消息
结果:   Agent 对 "之前" 的理解变得模糊
```

**缓解策略：**
- 压缩 Prompt 中明确要求保留用户偏好
- 保留较多的最近消息（牺牲压缩比换取连续性）
- 允许 Agent 在不确定时主动询问用户
- 关键信息可以写入 System Prompt 或外部文件（如 `CLAUDE.md`）

## 5. 会话管理的挑战

### 5.1 长对话的 Token 成本

Token 成本是一个经常被低估的问题：

```
假设:
- 平均每轮对话（请求 + 响应）消耗 5K tokens
- 一个复杂任务需要 30 轮对话
- 每次 API 调用发送完整历史

累计发送的 tokens:
  第 1 轮:  5K
  第 2 轮:  10K
  第 3 轮:  15K
  ...
  第 30 轮: 150K
  ──────────────
  总计:     ∑(5K×n) = 5K × 30 × 31 / 2 = 2,325K tokens

即使压缩将历史控制在 50K:
  压缩后第 20-30 轮: 50K + 5K×n = 55K ~ 100K
  节省了大约 40-60% 的 token 成本
```

**成本控制策略：**
- 及时压缩，避免发送过长的历史
- 对工具输出设置最大长度限制
- 使用 token 计数监控，给用户可见的反馈

### 5.2 压缩后上下文丢失的问题

这是一个根本性的挑战 —— **压缩是有损的**：

```
信息保真度
│
│  ████  100% (原始对话)
│  ███
│  ██    ← 压缩后 (可能只保留 60-80% 的关键信息)
│  █
│  ▒     ← 多次压缩后 (信息进一步衰减)
│
└────────────── 压缩次数 →
```

**实际表现：**
- 第 1 次压缩：通常影响不大，Agent 仍能很好地继续工作
- 第 2 次压缩：开始出现细节丢失，可能需要重新读取文件
- 第 3+ 次压缩：可能出现"摘要的摘要"，关键信息衰减明显

**应对方式：**
- 对于超长任务，考虑拆分为多个独立的子任务
- 将关键信息持久化到外部（文件、配置），而非仅靠对话上下文
- 在 `CLAUDE.md` 等项目级配置中记录重要约束

### 5.3 多会话间的隔离

当用户同时处理多个任务时，会话间必须严格隔离：

```
Session A: 重构模块 X        Session B: 修复 Bug Y
├── 读取了 fileX.ts         ├── 读取了 fileY.ts
├── 修改方案: 使用策略模式   ├── 定位到问题: 空指针
└── 状态: 进行中             └── 状态: 等待确认

隔离要求:
- Session A 不能访问 Session B 的上下文
- 切换会话时必须完整加载目标会话的状态
- 两个会话可能修改同一个文件（需要用户注意冲突）
```

**隔离的实现：**
- 每个会话独立的消息历史
- 独立的 token 计数和压缩状态
- 独立的工作目录配置
- 文件级别的锁（防止两个会话同时写同一文件）

## 6. 方案对比：上下文管理的不同策略

### 6.1 滑动窗口（Sliding Window）

```
原理: 只保留最近的 N 条消息，丢弃更早的消息

消息流:  M1  M2  M3  M4  M5  M6  M7  M8
窗口 (N=4):              [M5  M6  M7  M8]
丢弃:    M1  M2  M3  M4

优点: 实现简单，无额外 API 调用
缺点: 早期上下文完全丢失，无任何摘要
适用: 简单聊天场景，上下文依赖性低
```

### 6.2 向量数据库检索（RAG）

```
原理: 将历史消息向量化存入数据库，每轮检索最相关的片段

                    ┌─────────────┐
用户新消息 ────────→│  向量检索    │────→ 相关历史片段
                    │  (Embedding) │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  向量数据库  │
                    │  M1 M2 ... Mn│
                    └─────────────┘

优点: 能跨越很长的历史，精准召回相关信息
缺点:
  - 依赖 Embedding 质量，可能召回不相关内容
  - 丢失时序信息（不知道事件发生的先后顺序）
  - 需要额外的基础设施（向量数据库）
  - 对于 Agent 场景，"相关性"很难准确定义
适用: 知识库问答、文档检索场景
```

### 6.3 摘要压缩（Claude Code 使用的方式）

```
原理: 让 LLM 总结早期对话，保留摘要 + 最近原始消息

压缩前:  [System] [M1] [M2] ... [M15] [M16] [M17] [M18] [M19] [M20]
压缩后:  [System] [Summary of M1-M15] [M16] [M17] [M18] [M19] [M20]

优点:
  - 保留了历史的语义信息（比滑动窗口好）
  - 保留了最近对话的完整上下文（比纯 RAG 好）
  - 无需额外基础设施
  - LLM 自身理解什么信息重要

缺点:
  - 压缩本身需要一次 API 调用（额外成本）
  - 摘要质量依赖 LLM 能力
  - 信息仍然有损
  - 多次压缩会导致"摘要的摘要"

适用: Agent 场景，需要平衡上下文保持和容量限制
```

### 6.4 方案对比总结

| 维度 | 滑动窗口 | RAG 检索 | 摘要压缩 |
|------|---------|---------|---------|
| **实现复杂度** | 低 | 高 | 中 |
| **信息保留度** | 低（丢弃即消失） | 中（依赖检索质量） | 中高（语义级保留） |
| **时序保持** | 好（最近的完整） | 差（片段化） | 好（摘要有时序） |
| **额外成本** | 无 | Embedding 计算 | 一次 LLM 调用 |
| **基础设施** | 无 | 向量数据库 | 无 |
| **Agent 适配性** | 差 | 中 | 好 |
| **可预测性** | 高（确定性丢弃） | 低（检索不确定） | 中（摘要质量不确定） |

**为什么 Claude Code 选择摘要压缩？**

1. **Agent 的对话具有强时序性** —— 操作顺序很重要，滑动窗口和 RAG 都不能很好地保持
2. **CLI 工具需要零依赖** —— 不能要求用户安装向量数据库
3. **LLM 自身最懂什么信息重要** —— 让同一个模型来做摘要，比规则化方法更智能
4. **保留最近的完整上下文** —— 当前正在进行的操作需要完整的细节

## 总结

会话管理是 Agent 系统的"记忆层"，直接决定了 Agent 能否在长任务中保持连贯性和正确性。核心设计决策包括：

- **持久化**：本地文件系统，JSON 格式，零依赖
- **恢复机制**：resume / continue / fork / teleport，覆盖各种使用场景
- **压缩策略**：基于 LLM 的摘要压缩，在信息保留和容量限制间取得平衡
- **边界标记**：compact_boundary 系统消息清晰分隔压缩摘要和原始对话

最大的挑战仍然是 **有损压缩带来的信息衰减** —— 这不是技术问题，而是 LLM 有限 context window 带来的根本性约束。在这个约束被突破之前，优秀的压缩策略和外部持久化（如 `CLAUDE.md`、项目文档）仍是最实际的应对手段。
